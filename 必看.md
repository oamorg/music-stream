# 音乐流媒体系统 MVP 开发规范

本文件是项目 MVP 阶段的唯一执行基线。后续开发、任务拆分、代码生成、评审和验收，均以本文件为准。

适用对象：
- 产品/项目负责人：用它拆分迭代和验收
- 开发者：按模块边界和阶段顺序实现
- AI 工具：按本文约束生成代码，不得自行扩大范围

---

## 1. 文档目标

本规范只解决一件事：在最短可控周期内，交付一个能真实运行的音乐播放 MVP。

MVP 必须支持以下最小闭环：
- 用户注册、登录
- 音乐元数据录入
- 音频文件转码为 HLS
- 用户搜索歌曲
- 用户发起播放并通过权限校验
- 客户端上报播放事件
- 用户查看最近播放历史

本阶段不追求“大而全”，不以“未来可能需要”为理由提前引入复杂基础设施。

## 2. MVP 范围

### 2.1 必须交付的能力

1. 用户认证
   - 邮箱注册
   - 邮箱密码登录
   - JWT 鉴权
   - 刷新令牌

2. 曲库管理
   - 歌曲元数据录入
   - 音频文件上传到对象存储
   - 音频转码为 HLS
   - 歌曲发布状态管理

3. 搜索与播放
   - 按歌曲名、歌手名搜索
   - 查询歌曲详情
   - 播放前执行权限校验
   - 生成带时效的 HLS 播放地址

4. 行为记录
   - 播放开始事件
   - 心跳事件
   - 播放完成事件
   - 最近播放历史查询

5. 基础运维能力
   - 健康检查
   - 结构化日志
   - 基础指标
   - 本地和测试环境一键启动

### 2.2 明确不做的内容

以下内容不属于 MVP，禁止提前实现：
- 实时推荐系统
- 机器学习模型训练
- Kafka
- Elasticsearch
- Kubernetes
- 多区域部署
- 全球 CDN 调度
- 在线支付系统
- 社交、评论、点赞、歌单协作
- DRM 和复杂版权分区策略
- 多码率自适应流

如确需加入，必须先更新本文件，再开始实现。

### 2.3 MVP 成功标准

| 项目 | 目标 |
| :--- | :--- |
| 登录接口 P95 | 小于 300ms |
| 歌曲详情/搜索接口 P95 | 小于 300ms |
| 创建播放会话接口 P95 | 小于 500ms |
| 播放事件写入接口 P95 | 小于 200ms |
| HLS 转码成功率 | 大于等于 95% |
| 最近播放历史可见延迟 | 小于 1 分钟 |
| 单实例稳定并发能力 | 200 并发连接下无明显错误峰值 |
| 环境启动方式 | `docker compose up` 后可完成联调 |

说明：
- 本阶段目标是“可稳定交付”，不是“10k QPS 生产级极限性能”。
- 性能目标按单区域、单环境、单主实例思路评估。

## 3. 技术决策

### 3.1 总体原则

MVP 采用“单仓库、模块化单体、必要时异步任务”的方案，不做微服务拆分。

原因：
- 当前首要问题是交付闭环，不是组织级扩展
- 微服务会放大部署、联调、观测、故障定位成本
- 单体在边界清晰的前提下，后续仍可按模块拆服务

### 3.2 固定技术栈

| 模块 | 技术选择 | 说明 |
| :--- | :--- | :--- |
| 主后端 | Go 1.22+ | 统一后端语言，降低维护和 AI 生成复杂度 |
| HTTP 框架 | Gin | 轻量、成熟、足够支撑 MVP |
| 数据库 | PostgreSQL 16 | 主存储，负责事务和核心数据一致性 |
| 搜索 | PostgreSQL Full Text + `pg_trgm` | MVP 不引入 Elasticsearch |
| 对象存储 | MinIO（本地）/ S3 兼容服务（线上） | 存放原始音频和 HLS 文件 |
| 缓存 | Redis 7 | 只用于短期缓存、限流、会话辅助 |
| 异步任务 | PostgreSQL Outbox + Worker | MVP 不引入 Kafka |
| 音频处理 | FFmpeg | 负责转码和 HLS 切片 |
| 部署 | Docker Compose | 本地、测试环境统一 |
| 指标 | Prometheus | 暴露基础 metrics |
| 日志 | JSON 结构化日志 | 便于检索和排障 |

### 3.3 关键取舍

1. 只支持 HLS，不支持 DASH
   - HLS 生态更成熟，浏览器和客户端适配更直接

2. 只支持单码率输出
   - 默认 AAC 128 kbps
   - 多码率自适应播放留到后续阶段

3. 搜索先用 PostgreSQL
   - 只有当搜索召回、排序、规模明确不足时，才升级到 Elasticsearch

4. 异步事件先用 Outbox
   - 业务先写 PostgreSQL
   - 再由 Worker 可靠消费 Outbox
   - 只有明确出现跨服务高吞吐需求时才引入 Kafka

## 4. 系统边界与模块划分

### 4.1 模块列表

1. `auth`
   - 用户注册、登录、JWT、刷新令牌

2. `catalog`
   - 歌曲元数据管理
   - 歌曲发布状态
   - 搜索和详情查询

3. `media`
   - 原始音频上传
   - FFmpeg 转码
   - HLS 文件写入对象存储

4. `playback`
   - 播放权限校验
   - 创建播放会话
   - 生成签名播放地址

5. `history`
   - 播放事件接收
   - 最近播放历史查询

6. `platform`
   - 配置加载
   - 日志
   - 指标
   - 数据库
   - Redis
   - 对象存储

### 4.2 模块协作规则

- Handler 只能调用 Service，不能直接访问数据库
- Service 只能通过 Repository 访问数据库
- 模块之间只能通过公开 Service 接口交互
- 禁止一个模块直接读写另一个模块的表结构细节
- 平台能力必须下沉到 `platform`，不得在业务模块中重复实现

- `user_entitlements` 用于表示用户是否有播放权限

### 5.3 播放与历史

3. `outbox_events`
   - `id`
   - `event_type`

### 5.4 状态枚举

必须统一约束以下状态值：

- `tracks.status`
  - `DRAFT`
  - `PROCESSING`
  - `READY`
  - `BLOCKED`

- `track_assets.status`
  - `UPLOADED`
  - `TRANSCODING`
  - `READY`
  - `FAILED`

- `users.status`
  - `ACTIVE`
  - `DISABLED`

- `play_events.event_type`
  - `START`
  - `HEARTBEAT`
  - `COMPLETE`

## 6. 外部接口最小集合

MVP 对外只开放以下接口。新增接口前，必须说明它属于哪个用户路径。

### 6.1 认证接口

- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`
- `POST /api/v1/auth/logout`

### 6.2 曲库接口

- `GET /api/v1/tracks`
- `GET /api/v1/tracks/{trackId}`
- `GET /api/v1/search?q=`

### 6.3 播放接口

- `POST /api/v1/playback/sessions`
  - 输入：`trackId`
  - 输出：`sessionId`、`manifestUrl`、`expiresAt`

- `POST /api/v1/playback/events`
  - 输入：`sessionId`、`eventType`、`positionSec`、`clientTimestamp`

### 6.4 历史接口

- `GET /api/v1/me/history`

### 6.5 管理能力

MVP 不单独开发后台管理前端。

以下能力允许通过内部 CLI 或受保护的管理接口完成：
- 导入歌曲元数据
- 上传音频源文件
- 触发转码任务
- 写入用户播放权限

如果采用管理接口，其路由必须置于 `/internal/admin/*`，并且不能暴露给公网。

## 7. 关键业务流程

### 7.1 歌曲导入流程

1. 录入 `tracks` 基础信息，状态为 `DRAFT`
2. 上传原始音频到对象存储
3. 写入 `track_assets`，状态为 `UPLOADED`
4. 创建一条转码任务到 `outbox_events`
5. Worker 拉取任务，使用 FFmpeg 转码为 HLS
6. 转码成功后更新 `track_assets.status = READY`
7. 更新 `tracks.status = READY`
8. 任一步骤失败，记录错误并标记 `FAILED` 或 `BLOCKED`

### 7.2 播放流程

1. 客户端调用创建播放会话接口
2. 服务校验用户身份
3. 服务校验歌曲是否 `READY`
4. 服务校验用户是否拥有播放权限
5. 服务生成短时效签名 `manifestUrl`
6. 服务写入 `playback_sessions`
7. 客户端开始播放 HLS
8. 客户端定期上报播放事件

### 7.3 历史构建流程

1. `play_events` 接收 `START`、`HEARTBEAT`、`COMPLETE`
2. Worker 异步消费事件，生成最近播放历史
3. `/api/v1/me/history` 只返回用户自身数据

说明：
- MVP 可直接从 `play_events` 查询最近历史
- 当查询性能不足时，再补充单独的 `listening_history` 投影表

## 8. 迭代顺序与验收标准

必须严格按顺序推进。前一阶段未完成验收，不得开始后一阶段。

### Phase 0：工程初始化

目标：
- 搭建仓库骨架
- 接通基础依赖
- 保证可本地运行

必须完成：
- 初始化 Go 项目和目录结构
- 建立 PostgreSQL、Redis、MinIO 的 Docker Compose
- 建立数据库迁移机制
- 提供 `/health/live` 和 `/health/ready`
- 接入结构化日志和基础 metrics

验收标准：
- 新人拉代码后可在 30 分钟内跑通环境
- `docker compose up` 后，API 服务可成功启动
- 健康检查接口返回 200

### Phase 1：认证闭环

目标：
- 完成用户注册、登录、刷新令牌

必须完成：
- `users`、`refresh_tokens` 表迁移
- 密码哈希存储，禁止明文
- JWT 鉴权中间件
- 注册、登录、刷新、登出接口

验收标准：
- 正常注册和登录可以获得 access token 与 refresh token
- 错误密码返回 401
- 已登出 refresh token 不可再用
- 关键路径具备单元测试

### Phase 2：曲库与媒体处理

目标：
- 完成歌曲导入与 HLS 转码

必须完成：
- `tracks`、`track_assets` 表迁移
- 内部导入 CLI 或管理接口
- 对象存储上传
- FFmpeg 转码到 HLS
- 资产状态更新

验收标准：
- 任意一首测试音频可从源文件转为 HLS
- 成功后 `tracks.status = READY`
- 失败时可在日志和表中看到明确原因

### Phase 3：搜索与播放权限

目标：
- 完成搜索、详情、播放会话创建

必须完成：
- `GET /api/v1/tracks`
- `GET /api/v1/tracks/{trackId}`
- `GET /api/v1/search`
- `user_entitlements` 表和权限校验逻辑
- 创建播放会话并生成签名地址

验收标准：
- 仅 `READY` 状态歌曲可被搜索和播放
- 无权限用户请求播放返回 403
- 有权限用户可拿到可用的 `manifestUrl`

### Phase 4：事件与历史

目标：
- 记录播放行为并提供历史查询

必须完成：
- `playback_sessions`、`play_events`、`outbox_events` 表迁移
- 播放事件上报接口
- Worker 消费和历史查询逻辑
- `/api/v1/me/history`

验收标准：
- `START`、`HEARTBEAT`、`COMPLETE` 事件可落库
- 历史接口可返回最近播放记录
- 查询结果只包含当前用户数据

### Phase 5：稳定性与交付

目标：
- 达到可演示、可联调、可回归状态

必须完成：
- OpenAPI 文档
- 关键路径集成测试
- 限流与基础安全配置
- 指标埋点
- 错误预算和已知限制记录

验收标准：
- 登录、搜索、播放、事件上报链路可完整回归
- 基础压测下没有明显资源泄漏
- README 可指导完成部署和联调

## 10. 编码规范

### 10.1 命名

- 包名使用简短小写英文
- 导出类型和方法使用 Go 标准命名
- API 路径统一使用复数资源名
- 数据表名统一使用复数下划线风格

### 10.2 分层

- `handler`：参数解析、鉴权、响应格式
- `service`：业务编排、事务边界
- `repository`：SQL 和持久化
- `model`：领域对象和 DTO

禁止事项：
- Handler 中写 SQL
- Repository 中写跨模块业务判断
- 为了“方便”跳过 Service 直接访问 Repository

### 10.3 错误处理

所有错误必须满足以下流程：
- 捕获错误
- 记录结构化日志
- 包装为统一业务错误码
- 返回明确 HTTP 状态码

统一要求：
- 4xx 表示客户端输入、权限、资源状态问题
- 5xx 表示系统内部异常
- 禁止吞错
- 禁止空 `recover`
- 禁止只打印字符串不带上下文

### 10.4 配置管理

- 所有配置来自环境变量或配置文件
- 严禁把密钥写进代码仓库
- 必须提供 `.env.example`

### 10.5 数据库规范

- 任何表结构变更都必须有 migration
- 写操作默认在事务中处理
- 禁止跨模块共享 Repository
- 索引必须与真实查询场景对应，不准“先建一堆再说”

### 10.6 搜索规范

- 只对 `READY` 歌曲建立查询结果
- 按标题、歌手、专辑做全文或模糊检索
- 排序默认按相关度，其次按最近发布时间

### 10.7 播放规范

- 播放地址必须带有效期
- 默认有效期 10 分钟
- 过期后必须重新申请播放会话
- 无权限用户不得直接拿到对象存储真实路径

### 10.8 日志与指标

每个请求至少记录：
- `request_id`
- `user_id`
- `path`
- `status_code`
- `latency_ms`

必须暴露的指标：
- HTTP 请求总数
- HTTP 延迟直方图
- 数据库连接池状态
- 转码任务成功/失败计数
- 播放事件写入计数

## 11. 测试规范

### 11.1 必须覆盖的测试

1. 单元测试
   - 认证服务
   - 权限校验
   - 播放会话创建
   - 播放事件校验

2. 集成测试
   - 登录接口
   - 搜索接口
   - 创建播放会话接口
   - 播放事件写入接口

3. 媒体处理验证
   - 使用固定测试音频验证 FFmpeg 输出的 HLS 文件存在且可读取

### 11.2 不允许的情况

- 只写 Happy Path
- 修改接口却不更新测试
- 依赖人工点点点验证替代自动化回归

### 11.3 交付门槛

满足以下条件才算可合并：
- 相关模块单元测试通过
- 关键集成测试通过
- migration 可重复执行
- OpenAPI 已更新
- 无新增高危 TODO

## 12. 安全与权限基线

MVP 必须做到以下最低要求：

- 密码使用强哈希算法存储
- access token 与 refresh token 分离
- 刷新令牌支持失效和撤销
- 管理接口不得暴露公网
- 播放地址带时效签名
- 基础限流至少作用于登录接口和播放事件接口

MVP 明确不做：
- DRM
- 设备级授权体系
- 复杂地域版权策略

## 13. AI 工具使用规则

AI 生成代码前，必须先明确当前任务属于哪个模块和哪个阶段。

固定提示模板：

> 你现在负责本项目 MVP 的 `<模块名>` 模块开发，请严格遵守《音乐流媒体系统 MVP 开发规范》。不要新增未批准的中间件、服务或基础设施。任何接口变更都必须同步测试、迁移或文档。

AI 输出后的人工检查点：
- 是否违反了模块边界
- 是否引入了未批准的依赖
- 是否补齐测试
- 是否更新 OpenAPI 或 migration
- 是否把未来需求提前做进来

AI 明确禁止事项：
- 自行拆分微服务
- 自行引入 Kafka、ES、K8s
- 自行增加“推荐、支付、评论”等非 MVP 功能
- 在未说明的情况下改写已有接口语义

---

## 14. Definition of Done

一个任务只有同时满足以下条件，才算完成：

- 代码已实现
- 测试已补齐并通过
- 日志和错误码可观测
- 文档已同步更新
- 本地可复现
- 不引入超出阶段目标的新范围

---

## 15. MVP 之后的升级路线

只有在 MVP 稳定后，才允许按顺序评估以下升级：

1. 搜索升级到 Elasticsearch
2. 异步链路从 Outbox 升级到 Kafka
3. 增加推荐服务
4. 支持多码率 HLS
5. 引入 CDN
6. 按模块拆分服务
7. 引入 Kubernetes

顺序不可颠倒。先证明现有瓶颈，再增加复杂度。

---

## 16. 执行要求

项目启动后，默认按以下顺序创建任务：

1. 工程初始化与本地环境
2. 数据库迁移机制
3. 认证模块
4. 曲库与媒体处理模块
5. 搜索与播放模块
6. 行为事件与历史模块
7. 测试、文档、交付收尾

任何开发任务如果无法明确归入以上顺序之一，默认说明它不属于当前 MVP。
